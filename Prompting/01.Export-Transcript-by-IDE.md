## 프로젝트: Unit Converter

현재 워크스페이스(`UnitConverter_04`)는 **길이 단위 변환기** 프로젝트입니다.

### 현재 코드 상태

`UnitConverter.py`는 기본 동작만 있는 초기 버전입니다.

- `meter`, `feet`, `yard` 3개 단위 지원
- meter를 기준으로 변환
- 입력 형식: `meter:2.5`
- OCP/SRP, 테스트, 설정 외부화 등은 **아직 미구현**

### 구현해야 할 요구사항 (README 기준)

**기본**
1. `단위:값` 입력 → 다른 모든 단위로 변환 출력
2. 새 단위 추가 시 기존 코드 변경 최소화 (OCP)
3. SRP에 맞는 클래스 구조
4. 입력 검증 (음수, 잘못된 형식, 없는 단위)
5. 단위 변환 테스트 코드

**추가**
1. 변환 비율을 JSON/YAML 등 외부 설정에서 로드
2. 런타임 단위 등록 (예: `1 cubit = 0.4572 meter`)
3. 출력 포맷 선택 (JSON / CSV / 표)

### 권장 진행 순서 (Activities 6시간)

1. **0.5h** — 코드·요구사항 분석  
2. **2h** — OCP/SRP 설계, 입력 검증  
3. **0.5h** — 테스트 코드  
4. **2h** — 추가 요구사항 + 테스트  
5. **1h** — 회고 및 발표 (권용환님 발표)

### Dual-Track RED 테스트 설계

**Track A (UI/Boundary)** — `test_cli.py`
- U-IN-01: 빈 입력 → 형식 오류
- U-IN-02: `meter` (콜론 없음) → 형식 오류
- U-IN-03: `meter:-1` → 음수 거부
- U-OUT-01: `meter:2.5` → 3줄 이상 출력

**Track B (Domain/Logic)** — `test_converter.py`
- D-CNV-01: 1 feet → 0.3048 m
- D-CNV-02: 2.5 m → 8.20210 ft (5자리)
- D-CNV-03: feet→yard, meter 경유 일치
- D-REG-01: cubit 0.4572 등록 후 변환
- D-CFG-01: 깨진 JSON → ConfigError

**RED 단계 금지 규칙**
- RED에서 **구현 코드 작성 금지**
- `pytest.fail("RED: ...")` 허용
- skip / xfail 금지
- **1 RED 묶음 = 1 커밋**

## 3. 1일차 교재 (`응용-D1_...-교재.pdf`)

**핵심 개념 — C2C (Concept to Code)**
- 요구·개념 → 테스트 → 코드까지 **추적 가능**해야 함
- **ARRR ≡ Dual-Track TDD** (같은 것의 다른 이름)
  - Ask = RED
  - Respond = GREEN
  - Refine = REFACTOR
  - Repeat = 다음 질문

**바이브 코딩 vs C2C**
| 바이브 | C2C |
|--------|-----|
| 동작하면 채택 | 개념→테스트→코드 연결 |
| 추적 불가 | 깨지면 어느 계층인지 즉시 파악 |
| AI 코드 맹신 | AI 출력을 **테스트로 검증** |

**Cursor 8계층**
Model → Agent → Harness → **Rule** → **Skill** → Tool/MCP → **Command** → Test/Review Loop

**7단계**
① 주제 → ② R-G-I-O 설계 → ③ Ask(RED) → ④ Respond(GREEN) → ⑤ Refine(REFACTOR) → ⑥ Repeat → ⑦ 확장

**Ask vs Agent 모드**
- **Ask:** 분석·설명만 (코드 수정 없음)
- **Agent:** 파일 생성·수정·명령 실행

## 4. 패키지 설치

실습에 필요한 최소 패키지:

```bash
pip install --upgrade pip
pip install pytest
```

추가 요구사항(설정 파일 YAML)까지 미리 준비하려면:

```bash
pip install pytest pyyaml
```

팀 협업을 위해 `requirements.txt`를 만들어 두는 것을 권장합니다:

```text
pytest>=8.0
pyyaml>=6.0
```

생성 후:

```bash
pip install -r requirements.txt
```

강의 프로젝트 가이드에 **Python 3.11+** 가 적혀 있지만, 그건 권장 사양에 가깝고, README·시드 코드·목표 아키텍처를 보면 **3.11 전용 기능을 꼭 써야 하는 요구는 없습니다.**

### 3.10.11로 충분한 이유

- `venv`, `pytest`, `json`, 클래스/모듈 분리 → 3.10에서 모두 지원
- OCP/SRP 설계, 입력 파싱, 단위 변환, CLI → 표준 Python으로 구현 가능
- 타입 힌트(`list[str]`, `X | Y` 등)도 **3.10부터** 사용 가능
- 추가 요구사항(YAML)은 `pyyaml` 패키지로 해결 (Python 버전과 무관)

### 알아두면 좋은 차이 (실습에 큰 영향은 거의 없음)

| 항목 | 3.10 | 3.11+ |
|------|------|-------|
| 실행 속도 | 조금 느림 | 더 빠름 (체감은 테스트 많을 때) |
| `tomllib` (표준 TOML) | 없음 | 있음 → 이 프로젝트는 JSON/YAML 위주 |
| `typing.Self` | 3.11+ | 타입 힌트용, 없어도 구현 가능 |
| Exception groups 등 | 3.11+ | 이 과제 범위 밖 |

### 주의할 점 하나

- **팀원·강사 PC가 3.11** 이면, AI나 예제 코드가 가끔 3.11 문법/라이브러리를 제안할 수 있습니다.
- 그때는 “3.10 호환”이라고 Agent/Chat에 알려두면 됩니다.
- 가능하면 팀 내에서 **같은 major.minor(예: 전원 3.10.x 또는 3.11.x)** 를 맞추는 편이 좋습니다.

## Mom Test가 뭔가요?

**Mom Test**는 로브 피츠패트릭(Rob Fitzpatrick)의 고객·문제 검증 방법입니다.  
핵심은 **“이런 앱 만들 거예요”라고 말하지 말고, 상대(또는 본인)의 과거 경험만 듣는다**는 것입니다.

교재(`응용-D1_...-교재.pdf` 세션 3)의 **3원칙**:

| 원칙 | 하지 말 것 | 대신 할 것 |
|------|-----------|------------|
| ① | “단위 변환기 만들 거야” (내 아이디어 설명) | 문제 상황만 탐색 |
| ② | “좋을 것 같아요” 같은 추상적 질문 | **과거·구체·사실** — “그때 뭐가 귀찮았어?” |
| ③ | 칭찬·미래 의견 수용 | “그때 10분 걸렸어요” 같은 **실제 행동·비용**만 기록 |

교재 예시 한 문장 결과:
> “사람이 10개 선의 합을 수동 확인하느라 생기는 시간·실수 문제를, 같은 기준으로 반복 판정할 수 있게 한다.”

**Cursor를 인터뷰어로 쓰고, 전 과정 Ask 모드**로 진행하라고 되어 있습니다.

## 인터뷰 후 PRD까지의 전체 과정

강의 흐름을 시간순으로 보면 다음과 같습니다.

```
Mom Test 인터뷰
    ↓
한 문장 문제 정의 (① 주제)
    ↓
R-G-I-O 설계 (②)
    ↓
PRD §1~§8 작성 (docs/PRD = SSOT)
    ↓
PRD → To-Do → Test Case (C2C 추적)
    ↓
ARRR (RED → GREEN → REFACTOR)
```

### STEP 1 — Mom Test 인터뷰 (Ask 모드)

**목표:** “만들 앱”이 아니라 **진짜 문제**를 한 문장으로 뽑기.

**좋은 주제(=PRD 배경) 조건** (교재):
- 명확한 **Input**
- 예측 가능한 **Output**
- 바로 코드/프롬프트로 만들 수 있음
- 현업에 바로 쓸 수 있음

**UnitConverter에 적용 예시 질문** (Cursor에게 인터뷰어 역할 요청):

- “길이·단위를 바꿀 때 실제로 어떤 상황이었나요?”
- “그때 수동으로 계산하거나 검색했나요? 몇 분 걸렸나요?”
- “실수해서 다시 한 적 있나요? 어떤 단위였나요?”
- “엑셀/계산기/구글 중 뭘 썼나요? 왜 그게 불편했나요?”

**금지:** “단위 변환 CLI를 OCP로 만들 예정입니다” 같은 **솔루션 먼저 말하기**.

**산출물:** 팀이 합의한 **문제 한 문장** (① 주제)

예시 (팀마다 다름):
> “현장에서 m·ft·yd를 수동 환산할 때마다 비율을 외우거나 재검색하느라 시간과 오류가 발생한다.”

### STEP 2 — R-G-I-O 설계 (PRD 뼈대)

인터뷰로 문제가 잡히면, 교재 **STEP 2-a**로 넘어갑니다.  
**성공 기준은 나중에 테스트가 됩니다** (C2C 시작점).

| 요소 | UnitConverter 예시 |
|------|---------------------|
| **R** Role | “당신은 길이 단위 변환 CLI를 C2C로 구현하는 개발자입니다” |
| **G** Goal | PRD·테스트로 추적 가능한 단위 변환기 재구현 |
| **I** Input | `meter:2.5`, `feet:10`, 잘못된 입력 예시 3개 이상 |
| **O** Output | 전 단위 변환 결과 (표/JSON/CSV 등) |
| **성공 기준** | FR-01~05 통과, 음수·형식 오류 거부 등 (= 테스트 씨앗) |

**산출물:** R-G-I-O 표 (PRD에 들어갈 Goal·입출력·수용 기준의 초안)

---

### STEP 3 — PRD 본문 작성 (§1~§8)

교재 세션 5 **AI 기반 문서화**:
- **SSOT:** 요구사항은 `docs/PRD` **한 곳**에만
- 프롬프트 예: **「PRD §1~§8 구조로 작성해줘」**

자료에 §1~§8 제목 전체는 안 나와 있지만, Magic Square 예(`PRD §8.1`)와 프로젝트 가이드 FR/NFR 표를 합치면 대략 이렇게 구성하면 됩니다.

| 섹션 | 넣을 내용 |
|------|-----------|
| §1 | 개요 / 배경 (Mom Test에서 나온 **한 문장 문제**) |
| §2 | 목표 (C2C, 추적 가능한 재구현) |
| §3 | 사용자·사용 시나리오 |
| §4 | 입력·출력 규격 (`unit:value`, 출력 포맷) |
| §5 | 기능 요구 (FR-01~05) |
| §6 | 비기능 요구 (NFR-01 OCP, NFR-02 SRP, 검증) |
| §7 | 추가 요구 P1 (EXT-01~03: JSON/YAML, 동적 등록, format) |
| §8 | 상세 스펙·수용 기준 (비율, 예시 출력, 오류 조건) |

**UnitConverter 특이점:** `README.md`와 프로젝트 가이드에 **PRD 요약·FR/NFR 표가 이미 있음**.  
그래서 PRD 작성은 **처음부터 빈 종이**가 아니라:

1. Mom Test로 **왜 필요한지(문제)** 를 팀 언어로 쓰고  
2. 가이드의 FR/NFR/EXT를 **팀 PRD(`docs/PRD.md`)로 formalize**  
3. 이후 코드·테스트는 **이 PRD만 SSOT**로 추적  

하는 작업에 가깝습니다.

| FR-04 | 음수 | U-IN-03 |
| NFR-01 | OCP | D-REG + inch 추가 시나리오 |
| … | … | … |

**산출물:** 추적표 (PRD 부록 또는 별도 `docs/traceability.md`)

### STEP 5 — 그 다음이 코딩 (PRD 이후)

PRD가 잡힌 뒤 7단계 ③부터:

- **③ Ask (RED)** — Dual-Track 테스트 스켈레톤  
- **④ GREEN** — 최소 구현  
- **⑤ REFACTOR** — 구조 분리  
- **⑥ Repeat / ⑦ 확장** — EXT 요구  

첫 Cursor 프롬프트(프로젝트 가이드)도 **분석**이지 PRD 작성이 아닙니다:
> `@README.md @UnitConverter.py` 레거시 스멜과 **PRD 갭** 목록

→ **PRD를 먼저 팀 SSOT로 확정한 다음** 이 갭 분석을 하는 순서가 맞습니다.

## A팀 실무 체크리스트 (지금 할 일)

1. **Mom Test 인터뷰** (팀원 1~2명 또는 본인 경험, Cursor Ask 모드)  
2. **문제 한 문장** 팀 합의  
3. **R-G-I-O** 표 작성  
4. **`docs/PRD.md`** — §1~§8 + README/가이드 FR·NFR·EXT 반영  
5. **FR/NFR ↔ Test ID** 추적표  
6. (그 다음) 시드 코드 갭 분석 → RED 브랜치

## Cursor에 넣을 수 있는 프롬프트 예 (Ask 모드)

**인터뷰 (Mom Test):**
```
Mom Test 3원칙을 지키며 인터뷰어 역할을 해줘.
나는 솔루션(단위 변환 앱)을 말하지 않을게.
업무에서 길이/단위 환산이 필요했던 과거 경험만 질문해줘.
```

**PRD 작성 (인터뷰·R-G-I-O 결과 반영 후):**
```
아래 인터뷰 요약과 R-G-I-O, README.md 요구사항을 바탕으로
docs/PRD.md를 §1~§8 구조로 작성해줘.
FR-01~05, NFR-01~02, EXT-01~03와 테스트 ID 매핑 표를 포함해줘.
```

**요약:** 강사님 말씀대로 **첫 골은 “PRD 작성”이 맞고**, 그 전에 **Mom Test 방식 인터뷰**로 문제 한 문장을 뽑고 → **R-G-I-O** → **`docs/PRD` §1~§8** 순서입니다. UnitConverter는 주제가 정해져 있어도 **“왜 필요한가”(Mom Test)** 는 팀이 직접 채워 넣는 단계입니다.

인터뷰 질문 리스트를 UnitConverter용으로 더 구체화해 드릴까요, 아니면 §1~§8 PRD 목차 초안을 템플릿으로 적어 드릴까요?

## 진행 전에 알아둘 점

교재 원칙:
- **Ask 모드** (분석·대화만, 코드/파일 수정 없음)
- **솔루션 말하지 않기** — `@README.md`, `@UnitConverter.py`는 **Mom Test 중에는 붙이지 마세요** (이미 “단위 변환기”가 드러남)
- 목표 산출물: **문제 한 문장** → 이후 R-G-I-O → PRD §1~§8

Cursor가 양쪽 역할을 하게 하려면, 답변자 페르소나를 먼저 고정하는 것이 좋습니다.  
예: “현장에서 스펙·도면 단위를 자주 바꿔 쓰는 개발자” 또는 “본인(박교현)의 실제 경험을 바탕으로 한 페르소나”.

## 1단계: 세션 설정 프롬프트 (먼저 이것부터)

```
당신은 Mom Test 진행을 돕는 퍼실리테이터입니다.
교육 과정의 Mom Test 3원칙을 반드시 지켜주세요.

[Mom Test 3원칙]
1. 내 아이디어(만들 제품·앱·기능)를 말하지 않는다.
2. 과거·구체·사실만 묻고 답한다. ("최근에~", "그때 몇 분~", "무엇을 썼~")
3. 칭찬·미래 의견("좋을 것 같아요", "쓸 것 같아요")은 근거로 쓰지 않는다.

[진행 방식]
- 당신이 **인터뷰어(질문자)** 와 **답변자(인터뷰이)** 역할을 번갈아 수행합니다.
- 한 턴 = 인터뷰어 질문 1개 → 답변자 답변 1개.
- 총 8~10턴 진행한 뒤, Mom Test 원칙 위반 여부를 스스로 점검합니다.

[답변자 페르소나]
- 이름: 박교현
- 배경: 소프트웨어 개발자, Cursor로 코드 작성·분석·리팩토링 경험 있음
- 맥락: 업무/실습 중 "수치·단위·변환·검증"이 반복되는 일상적 불편을 과거 사실로 답함
- 답변자는 솔루션(앱, CLI, 자동화 도구)을 제안하지 않음

[금지]
- "단위 변환기", "UnitConverter", "CLI를 만들" 같은 표현 사용 금지
- 가설 검증 질문 금지 ("이 기능 있으면 쓰시겠어요?")
- 추상적 질문 금지 ("불편하셨나요?"만으로 끝내지 말고 "언제/어디서/몇 번/몇 분"으로 구체화)

[출력 형식]
각 턴마다 아래 형식만 사용:

---
턴 N
[인터뷰어] (질문)
[답변자] (답변)
---

준비되면 턴 1을 시작하세요. 첫 질문은 가장 최근의 구체적 경험부터 물어보세요.
```

## 2단계: 한 번에 끝내는 통합 프롬프트 (8~10턴 자동 진행)

```

역할: 당신이 인터뷰어와 답변자를 모두 연기합니다.
답변자 = 박교현 (개발자, Cursor 사용 경험 有)

Mom Test 3원칙:
(1) 제품 아이디어 언급 금지
(2) 과거·구체·사실만
(3) 칭찬·미래 의견 무시

인터뷰어는 아래 질문 축을 순서대로 깊게 파고듭니다:
A. 최근에 숫자/단위/환산 때문에 시간 쓴 구체적 상황 1건
B. 그때 어떤 도구를 썼는지 (엑셀, 계산기, 검색, 메모, 코드 직접 수정 등)
C. 걸린 시간, 횟수, 실수 여부
D. 실수했을 때 어떤 피해/재작업이 있었는지
E. 비슷한 일이 반복되는 패턴이 있는지
F. 그 일을 누가/어디서 요청했는지 (맥락)

8턴 대화 후 반드시 아래 산출물을 작성:

## 1. Mom Test 원칙 준수 점검
- 위반한 턴이 있으면 지적
- 사용된 사실 근거만 bullet로 정리

## 2. 문제 한 문장 (주제)
"~문제를, ~할 수 있게 한다" 형식, 솔루션 명시 없이

## 3. 좋은 주제 조건 체크
- 명확한 Input / 예측 가능한 Output / 즉시 실행 가능 / 현업 사용 가능 (각 O/X + 한 줄)

## 4. R-G-I-O 초안
- Role / Goal / Input(예시 3개) / Output / 성공 기준(테스트 씨앗 3개)

## 5. PRD 작성용 메모
- §1 배경에 넣을 사실 3개
- §5 기능 요구 후보 (아직 ID 없이 문장만)
- §6 비기능(신뢰성·검증·반복 작업) 후보

지금 턴 1부터 시작하세요.
```

```
산출물:
1) 사실 근거 목록 (5개 이내, 각각 "언제/어디/무엇/얼마나")
2) 문제 한 문장 (솔루션 단어 금지)
3) R-G-I-O 표
4) PRD §1(배경) 초안 5문장

검증: 문제 한 문장에 "앱/프로그램/자동화/변환기" 같은 단어가 있으면 다시 쓰세요.
```

## 4단계: 본인 경험을 섞는 하이브리드 (추천)

Cursor가 답변자까지 연기하면 **가짜 답**이 될 수 있습니다.  
교육에서는 보통 **인터뷰어=Cursor, 답변자=본인**이 더 Mom Test에 가깝습니다.  
양쪽 연기를 원하시면, 아래처럼 **사실 시드**를 주는 편이 낫습니다.

```
Mom Test를 진행합니다. 역할 분담:
- 인터뷰어: 당신 (Mom Test 3원칙 준수)
- 답변자: 아래 [사실 시드]만 근거로 답변. 시드에 없는 내용은 "모름"이라고 함.

[사실 시드 - 박교현]
- (여기에 본인 실제 경험 3~5줄 적기)
  예: "지난주 도면 ft 단위로 왔는데 내부는 m 기준이라 수동 환산함"
  예: "구글 검색으로 3.28084 확인하고 엑셀에 붙여넣음"
  예: "한 번 소수점 자리 잘못 넣어서 다시 계산함"

8턴 인터뷰 후 문제 한 문장 + R-G-I-O + PRD §1 초안을 작성하세요.
제품 아이디어(단위 변환기 등)는 인터뷰 전 과정에서 금지합니다.
```

## 인터뷰어가 물어볼 좋은 질문 예 (lecture 톤)

교재가 강조하는 **과거·구체·사실**형만 사용:

| 나쁜 질문 (금지) | 좋은 질문 (Mom Test) |
|------------------|----------------------|

| 이런 기능 있으면 쓸래요? | 비슷한 일이 **일주일에 몇 번** 정도 있나요? |

## Mom Test 대화 (8턴)

---

## 5. PRD 작성용 메모

**§1 배경에 넣을 사실 3개**
1. 해외 PDF feet/inch 12칸을 mm BOM으로 옮기는 데 **35~40분**, 검색·계산기·엑셀 반복
2. 단위 검색 착오(inch→cm) **2건** → BOM 수정·재출력·회의 전 **10분** 지연
3. 분기 **2~3회** 유사; 레거시 코드에 **환산 상수가 파일마다 다르게** 존재

**§5 기능 요구 후보 (문장만)**
- `단위:숫자` 형식 문자열을 파싱한다
- 한 입력값을 **지원하는 모든 길이 단위**로 환산해 보여준다
- 지원하지 않는 단위는 **명확한 오류**로 처리한다
- 음수·빈 값·콜론 없는 형식은 거부한다
- (확장) 설정 파일에서 단위·비율을 읽는다
- (확장) 새 단위·비율을 등록한 뒤 바로 환산에 반영한다
- (확장) 결과를 표·JSON·CSV 형태로 낼 수 있다

**§6 비기능 후보**
- **신뢰성:** meter 기준 비율 일관(예: 3.28084 / 1.09361), 재계산 시 동일 결과
- **검증:** 잘못된 형식·음수·미지 단위에 대한 입력 검증
- **유지보수(OCP/SRP):** 단위 추가 시 기존 환산 핵심 로직 변경 최소; 파싱·환산·출력·설정 로드 분리
- **추적성(C2C):** 위 요구마다 테스트 ID 1:1 매핑
